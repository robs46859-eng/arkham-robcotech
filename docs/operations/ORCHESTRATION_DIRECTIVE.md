# Orchestration Directive

Generated: 2026-04-29

## Purpose

This directive defines how orchestration is used in Arkham and where its ownership boundaries begin and end.

It is the project-level rule set for workflow execution.

## Directive

Orchestration is the execution control plane for Arkham.

It is authoritative for:

- workflow lifecycle
- task dispatch
- checkpointing
- retries
- queue draining
- worker coordination
- executor registration
- workflow state visibility

Primary code surface:

- `services/orchestration/app/main.py`
- `services/orchestration/app/worker.py`
- `services/orchestration/app/tasks/__init__.py`
- `services/orchestration/app/flows/registry.py`
- `services/orchestration/app/checkpoints.py`
- `services/orchestration/app/queues.py`

## Non-Negotiable Rules

1. No vertical service owns workflow engine semantics.
2. No vertical service implements its own checkpoint system.
3. No vertical service implements its own generic retry engine.
4. Workflow progress must be observable through orchestration-owned state.
5. Task execution must flow through registered executors.
6. Queue behavior must remain centralized and testable.
7. Mutating operational commands generated by planning agents must not bypass orchestration or explicit human approval rules.

## Ownership Boundaries

## Orchestration owns

- the definition of workflow state transitions
- dispatch of task types to executors
- retry and failure-handling semantics
- checkpoint persistence and resume behavior
- worker lifecycle behavior
- queue inspection and drain operations
- workflow status APIs

## Vertical services own

- domain inputs
- domain entities
- domain-side business rules
- domain-side task payload construction
- interpretation of workflow results

## Shared boundary

Vertical services may request execution.

Orchestration decides:

- how tasks are sequenced
- how failures are retried
- how progress is resumed
- how workers process queued work

## Required Design Pattern

Every substantial multi-step business behavior should follow this shape:

1. A service receives domain intent.
2. That service creates or requests a workflow.
3. Orchestration registers and dispatches task execution.
4. Executors perform bounded units of work.
5. State is checkpointed and observable.
6. Domain services consume results rather than silently embedding workflow logic.

## Media-Commerce Directive

`services/media-commerce` is the first hardening vertical and must follow these rules.

### Allowed ownership

- entities and domain models in `services/media-commerce/app/models/`
- domain-specific agent logic
- domain APIs
- domain event interpretation

### Forbidden ownership

- private workflow runners
- local retry engines
- local checkpoint stores
- platform-owned cache infrastructure
- platform-owned memory persistence
- shadow task registries

### Required execution seam

Media-commerce execution should route through orchestration task executors, especially:

- `services/orchestration/app/tasks/media_commerce.py`

That file is the correct place for workflow-step execution logic that belongs to the orchestration plane.

## Planning Agent Directive

The following packages are planning-only:

- `launch-agent`
- `finisher-agent`
- `azure-runner-agent`
- `omni-deployer`

They may:

- inspect
- plan
- summarize
- recommend
- stage human-approved next actions

They may not become the implicit workflow engine.

If planning agents need execution coordination, they should:

1. reference orchestration-owned flows
2. preserve explicit approval gates
3. avoid embedding production state transitions in agent prompts or ad hoc tool code

## Testing Directive

The minimum orchestration maintenance surface includes:

- worker lifecycle tests
- task executor tests
- checkpoint behavior tests
- empty-queue and multi-task drain behavior tests
- failure and retry tests

Priority test surfaces already indicated by the repository include:

- `services/orchestration/tests/test_worker.py`
- `services/orchestration/tests/test_media_commerce_executor.py`

When new executors are added, they should not ship without explicit test coverage.

## Operational Directive

For day-to-day maintenance, orchestration should be reviewed before any major vertical diagnosis when the symptom includes:

- stuck work
- duplicate work
- missing task completion
- partial execution
- out-of-order effects
- retry loops
- workflow state drift

In practice, orchestration is the first place to look when behavior spans multiple services.

## Documentation Directive

Any architecture or deployment document that describes execution behavior must align with this directive.

If another document implies that a vertical service owns execution lifecycle directly, that document is stale and should be corrected.

## Conclusion

Arkham should evolve as a platform where orchestration is the execution authority and verticals remain disciplined clients of that authority.

If there is uncertainty about whether logic belongs in a vertical service or orchestration, default to orchestration when the logic concerns:

- sequencing
- retries
- checkpoints
- task dispatch
- worker control
- resumability
